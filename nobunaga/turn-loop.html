The Main Turn Loop

vm_bootstrap $A778 — the per-season driver every other subsystem hangs off.
① decompiler → C ② data-walk labels ③ bytecode-verified control flow ⑤ the spine / map

This is the spine of the game. One routine — vm_bootstrap $A778 — sequences every season: it runs the per-season tick, reshuffles who acts, and hands each fief to the command driver, looping out to the combat engine whenever a war is declared. The command palette (21 pages) and the battle engine are its two big branches; this page is the index above them, and the status badges are the remaining-work map.

Phase 1 · bank 0

divedai_strategic_turn_planner $A455

The heart of each season. In order:

  • Season save ($A47C) — if sram_save_pending_flag is set, write_sram_save_checksum_and_signature commits SRAM.
  • Calendar advance ($A47F) — current_season = (season+1)&3; current_game_year increments only when that wraps to 0 → 4 seasons = 1 year.
  • Re-roll market rates ($A495) — the per-period price book (roll_period_rate_table_6e0b); buy-low / sell-high timing is real because of this.
  • The season-phased economy update (per_period_fief_daimyo_update_driver $A30D) — see the table below.
  • Random-event spawn — pick a disaster / build / ravage handler, roll ~3% per fief to build the affected-province list, run it (the cracked event-spawn formula).
  • Per-daimyo illness pass — mark weakness with probability (100 − health) / 400, then “… has taken ill.”
This single routine is a candidate to graduate to its own deep-dive page — it is the densest node in the loop.

Phase 2 · bank 0

divedspawn_zealot_uprising_force_from_province $A5E6

When an uprising fires, this constructs a full 26-byte province-record army in zealot_uprising_slot $7515 (so it drops straight into the combat engine with no special-casing): each stat field seeded as pct_op(rng%20+40, source_field), plus men/rice drawn from the host province. The marker values (header 9999, extreme loyalty/morale, tiny skill/arms) are what make it recognisable in a save dump.

Phase 3 · bank 2

Any battle staged this season (a war declared during the previous command pass, or the uprising above) is resolved by handing control to the bank-2 combat engine (battle_init_driver $AFE1): tactical map, the 4-quadrant {1,2,4,7} damage model, the 31-day commander-resolution loop, ravage & spoils. Phase 4 · bank 0

divedshuffle_fief_turn_order_array $A742

Each season the turn order is reshuffled: scenario_fief_count random transpositions (swap_byte(rng%count + $6F1B, rng%count + $6F1B)). “Random beats initiative” — there is no fixed seating; a fief’s position in the order is re-rolled every season. The array lives at daimyo_turn_order $6F1B (see the find below — it is not at $6F0B as the labels long claimed).

Phase 5 · bank 1

has pageai_per_fief_command_driver $B89B

Walks daimyo_turn_order in order; for each fief, selected_province_idx = daimyo_turn_order[ai_per_fief_loop_index], then dispatches on province_ai_state (the six Grant policies — Home / Industrial / Military / Balanced / Farming / Direct) into the matching AI econ handler, and finally issue_province_command executes it. Resting daimyo are skipped via rest_turns_remaining. This is the “command palette” branch — all 21 command pages are the leaves of this one node.

The mechanics it explains

Behaviours that had been hinted at across the project but never pinned to code, now ground-truth from the bytecode:

The command pass resumes across combat. ai_per_fief_loop_index $6001 is a persistent global, restored into the frame at entry ($B8A4). When a daimyo declares war mid-pass, the index is saved ($B996), control loops out through the bank-2 combat engine, then re-enters and continues at the next fief — the season’s command pass is interruptible and resumable, not restarted.
4 seasons = 1 year, and each season has a job. config_block $6D5F was a misnomer — at runtime it is the season counter (0–3), renamed current_season. It increments each period; the year ticks only on its wrap to 0. The switch on it is what gives each season a distinct economic phase (above) — the harvest is specifically a season-2 event, confirming the long-standing “fall harvest” reading from the other direction.
The turn order lives at $6F1B, not $6F0B. The daimyo_turn_order label had been mis-addressed by 16 bytes — $6F0B has zero references anywhere. The real array (shuffled by $A742, read by the command driver at $B918) is $6F1B. The label was relocated and a latent bug in sram-decode-province.py (which dumped the wrong address) was fixed — exactly the drift-as-re-derivation this project exists to kill.
