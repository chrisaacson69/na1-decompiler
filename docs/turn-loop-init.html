New Game

init_new_game_state $89A3 — the front-end wizard that builds a fresh game.
① decompiler → C ② data-walk labels ③ bytecode-verified flow ④ emulator-probed formula

Before the season loop can run, the board has to exist. init_new_game_state is the setup wizard: it boots the title, checks for a save, then walks the player through scenario, player count, daimyo creation, and skill level — seeding every fief, the turn order, and the market before handing off to the main loop. Most of it we’d glimpsed in earlier walks; reading it end-to-end turned up the answer to a question the game never explains: what does the skill level actually do?

The steps

The wizard, in order — each step grounded in the $89A3 bytecode:

  1. $89B1Boot & title. render_boot_title_screens plays the KOEI / Nobunaga title sequence; audio is reset (call_bank10_entry(0)).
  2. $89CCSave check. verify_sram_save_integrity — if a valid save is present the routine bails out here (the load path takes over). New-game proceeds only on no / invalid save.
  3. $89EBChoose the scenario. prompt_select_scenario_size — 17-fief or 50-fief — sets scenario_fief_count, which every later loop bound reads.
  4. $89F9Seed the board. daimyo_turn_order[i] = i (identity — later reshuffled each season); province_ai_state[i] = 0 (every fief starts under manual/AI-default control); the five market rates $6E0B..$6E13 are randomized to rng%10 + 8.
  5. $8A22How many players? newgame_player_count = number_input(1..8).
  6. $8A91Character creation (per human player). prompt_select_player_daimyo picks the house; daimyo_creation_stat_roll_screen rolls its stats and marks that fief province_ai_state = 5 (Direct / player-controlled) — the flag that later excludes it from the AI starting boost.
  7. $8AD6Skill level. “Please select skill level.” → const_two = number_input(1..5). This one input is the whole difficulty system (see below).
  8. $8ADFConfirm. “Is everything OK so far?” — yes proceeds; no loops all the way back to the scenario step and starts the selection over.
  9. $8B50Finalize. ai_player_count = 8 − humans; apply_scenario_starting_stat_boosts buffs the AI; “Then, on with the game!”

What skill level actually does

The skill level isn’t a separate difficulty system — it’s written straight into const_two ($6D63), a single value the engine reads in ~35 places across all four banks. The headline: all five development commands use (6 − skill) as the gain multiplier, so:

Grow gain = 2 · floor( amount · (6 − skill) / sqrt(output + amount) )

Emulator-confirmed on one fief (amount = 100), via tools/probe-skill-impact.py:

That also explains the mystery “×5” in every early Grow/Build derivation: those tests ran at skill 1, where 6 − 1 = 5. The cracked formulas were all implicitly the default-skill case.

One dial, three pressures

Picking a higher skill level doesn’t just make the AI “smarter” — the same value leans on three subsystems at once, all against the player:

PressureMechanismEffect of higher skill
Your economy(6 − skill) develop multiplierup to ~5× slower growth
AI head startapply_scenario_starting_stat_boosts: AI daimyo stats + skill×5, province fields + skill×10 — applied only to province_ai_state == 0 (i.e. not your Direct fiefs)stronger opponents from turn 1
World volatilityuprising size (skill+1)×10, combat casualty rolls rng(6 − skill) / −skill damagebigger rebellions, harsher battles

The boost loop’s province_ai_state == 0 gate is the key: character creation sets your own fief to state 5, so the buff lands on every AI house but never on you.

What it pinned down

Skill level = const_two, the economy throttle. “Please select skill level.” writes 1–5 to const_two — the very value the development formulas divide the field-root into. Difficulty is, first and foremost, an economy dial.
The starting boost targets the AI, confirmed. apply_scenario_starting_stat_boosts only buffs fiefs with province_ai_state == 0; your daimyo-creation step marks your fief state 5 (Direct), so you are excluded. Higher skill = bigger AI head start, never a player one.
